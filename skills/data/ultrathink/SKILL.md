# Ultrathink

複雑な問題に対して自動的に深い思考モードに切り替えます。

## 目的

- 複雑な問題を検出したら自動的に深思考モードを有効化
- ユーザーに確認せずシームレスに切り替え
- より深い分析と推論で問題を解決

## ワークフロー

### 1. 問題の複雑さを自動評価

ユーザーのリクエストを分析し、深い思考が必要かどうかを自動判断。

### 2. 自動切り替え

複雑な問題を検出したら、自動的に深思考モードに切り替え：

**深い思考が必要な問題（自動で有効化）:**
- 原因不明のバグ
- パフォーマンスボトルネック
- アーキテクチャの設計判断
- セキュリティの脆弱性分析
- 複雑なアルゴリズムの設計
- 技術的なトレードオフの評価
- 複数の仮説を検討する必要がある問題

**通常の思考で十分な問題（深思考モードは使用しない）:**
- 単純なバグ修正
- 定型的なコード生成
- 簡単な質問への回答
- ドキュメント更新

### 3. 深い思考を実行

自動的に以下のアプローチで分析：

1. **問題の分解**: 問題を小さな部分に分解
2. **仮説の生成**: 複数の仮説を立てる
3. **証拠の収集**: 各仮説を検証するための情報を収集
4. **仮説の評価**: 証拠に基づいて仮説を評価
5. **結論の導出**: 最も可能性の高い原因と解決策を特定

## 自動切り替えのトリガー

以下のような状況を検出したら自動的に深思考モードを有効化：

| トリガー | 例 |
|----------|-----|
| 原因不明の問題 | 「なぜこうなるのか分からない」「原因が特定できない」 |
| 設計判断 | 「どのアプローチが最適か」「アーキテクチャを決めたい」 |
| パフォーマンス | 「パフォーマンスが悪い理由」「最適化したい」 |
| セキュリティ | 「セキュリティリスクの評価」「脆弱性を確認」 |
| 複雑なロジック | 「複雑なアルゴリズム」「難しい実装」 |

## 出力形式

### 分析結果

```markdown
## 問題分析

### 仮説 1: {仮説}
- 証拠: {証拠}
- 可能性: 高/中/低

### 仮説 2: {仮説}
- 証拠: {証拠}
- 可能性: 高/中/低

## 結論

最も可能性の高い原因: {原因}

## 推奨される解決策

1. {解決策 1}
2. {解決策 2}
```

## 重要な注意事項

- ✅ 複雑な問題を検出したら自動的に深思考モードを有効化
- ✅ ユーザーに確認せずシームレスに切り替え
- ✅ 段階的に分析を進める
- ✅ 複数の仮説を検討
- ❌ 単純な問題に対して過剰に分析しない
- ❌ ユーザーの時間を無駄にしない
